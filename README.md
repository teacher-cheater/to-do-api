# To-Do List — Laravel API + Nuxt 3 SPA

Тестовое задание: список задач с авторизацией, CRUD и клиентской частью на
Vue/Nuxt поверх Laravel API.

```
todo-app/
├── backend/    Laravel 12 + Sanctum — API-код для вставки в свежий проект
└── frontend/   Nuxt 3 (Composition API) — SPA-клиент
```

## Быстрый старт — всё через Docker Compose (рекомендуется)

Единственное, что нужно на компьютере — **Docker Desktop**. Ни PHP, ни
Composer, ни Node ставить не нужно — всё это есть внутри контейнеров.

```bash
docker compose up --build
```

Первый запуск займёт несколько минут: соберутся образы, `backend`
скачает Laravel-скелет и зависимости через Composer, накатит миграции
и заполнит базу демо-данными, `frontend` поставит зависимости через
npm. Дальнейшие запуски (`docker compose up`, без `--build`) — быстрые,
всё уже установлено (хранится в именованных volume `backend_vendor` и
`frontend_node_modules`).

После того как в логах увидите `Starting Laravel dev server` и
`Nuxt dev server ready`:

| Сервис                        | Адрес                                      |
| ----------------------------- | ------------------------------------------ |
| Frontend (SPA)                | http://localhost:3000                      |
| Backend API                   | http://localhost:8000                      |
| phpMyAdmin                    | http://localhost:8080 (root / root_secret) |
| Swagger UI (документация API) | http://localhost:8081                      |

Откройте `http://localhost:3000`, войдите как `demo@example.com` / `password`.

```bash
docker compose logs -f backend    # если что-то не завелось — сюда
docker compose down               # остановить всё (данные останутся)
docker compose down -v            # остановить и стереть базу/зависимости полностью
---
docker compose down -v --remove-orphans #снести volume-ы
---
docker compose exec backend php artisan config:clear # закэшированный конфиг - очистка
docker compose exec backend php artisan cache:clear  # закэшированный конфиг - очистка
```

**Как это работает "под капотом":** `backend/` в этом репозитории
содержит не полный Laravel-проект, а только прикладной код (модели,
контроллеры, миграции и т.д.) — стандартные сгенерированные файлы
(`artisan`, `public/index.php`, `config/app.php` и подобные) в git не
хранятся, потому что не несут уникальной информации о проекте. При
первом запуске `docker-entrypoint.sh` внутри контейнера сам
догенерирует эти файлы через `composer create-project` во временную
папку и копирует недостающее в `/var/www` (примонтированную папку
`backend/`), **не трогая** уже существующие файлы — поэтому весь наш
код остаётся как есть. После этого локальная папка `backend/`
физически станет полноценным Laravel-проектом — это нормально, эти
файлы дальше коммитятся в git как обычно (см. `RUN_AND_PUSH_GUIDE.md`).

## Запуск без Docker

Если Docker не хочется — тот же результат получается вручную. Полная
пошаговая инструкция (что установить, что скопировать, чем проверить
каждый шаг) — в `docs/RUN_AND_PUSH_GUIDE.md`.

## Docker Compose: что именно поднимается и зачем

```yaml
services:
    mysql: # mysql:8.0, база "todo" создаётся из переменных окружения
    phpmyadmin: # GUI для базы — http://localhost:8080
    backend: # свой Dockerfile: PHP 8.2 CLI + Composer, php artisan serve
    frontend: # свой Dockerfile: Node 20, npm run dev
    swagger-ui: # рендерит docs/openapi.yaml — http://localhost:8081
```

**Как создаётся база.** Ничего вручную создавать не нужно — при первом
запуске контейнера `mysql` образ сам выполняет `CREATE DATABASE todo`
и заводит пользователя `todo`/`secret` с правами на неё (переменная
`MYSQL_DATABASE` в `docker-compose.yml`). Таблицы внутри неё создаёт
`docker-entrypoint.sh` командой `php artisan migrate` при старте
контейнера `backend` — он же дожидается, пока MySQL реально готов
принимать соединения (`healthcheck` + ручной retry-цикл в скрипте, на
случай если health-check зелёный на секунду раньше, чем сервер готов
к TCP-подключениям).

**Данные переживают перезапуск.** `docker compose down` останавливает
контейнеры, но volume `mysql_data` остаётся — при следующем
`docker compose up` база и все задачи будут на месте. Только
`docker compose down -v` стирает данные полностью (используйте, если
нужно начать с чистой базы и свежими demo-данными).

**Сиды запускаются один раз.** `docker-entrypoint.sh` проверяет,
пуста ли таблица `users`, прежде чем звать `db:seed` — иначе на каждый
рестарт контейнера попытка создать `demo@example.com` заново упала бы
на уникальном email.

**Команды на каждый день:**

```bash
docker compose up -d          # поднять всё в фоне
docker compose ps             # проверить статус/healthcheck
docker compose logs -f backend  # логи backend, если что-то не подключается
docker compose down           # остановить (данные в volume сохранятся)
docker compose down -v        # остановить и стереть всё (базу и node_modules/vendor) полностью
```

**Если порт занят** (3306, 8000, 3000, 8080 или 8081) — поменяйте
маппинг портов в `docker-compose.yml` (например, `"3307:3306"`); для
`mysql` дополнительно поправьте `DB_PORT` в `backend/.env` (актуально
только для запуска backend без Docker — см. ниже).

## Запуск backend без Docker (для справки)

Если backend нужно запустить на хосте напрямую (например, для
дебаггера, который плохо дружит с контейнерами) — MySQL всё ещё можно
брать из Docker:

```bash
docker compose up -d mysql
cd backend
composer create-project laravel/laravel /tmp/skel --no-install  # только для сравнения, обычно так делать не нужно
```

Полная версия этого пути — в `docs/RUN_AND_PUSH_GUIDE.md`, раздел
«Запуск без Docker».

## API-документация (Swagger / OpenAPI)

Полная спецификация — `docs/openapi.yaml` (OpenAPI 3.0), описывает все
эндпоинты из ТЗ: параметры, тела запросов, схемы `Task`/`User`, и все
коды ответов (`200/201/204/401/403/404/422`).

**Просмотр:**

```bash
docker compose up -d swagger-ui
# → http://localhost:8081
```

Swagger UI в контейнере просто рендерит статический YAML-файл — не
требует Laravel вообще, можно смотреть документацию, даже не поднимая
backend. Кнопка «Try it out» будет реально стучаться на
`http://localhost:8000`, но так как для авторизованных ручек нужен
заголовок `Authorization: Bearer <токен>` (получить его — через
`POST /api/auth/login` там же в Swagger UI, кнопка «Authorize»), для
«живого» тестирования запросов из браузера удобнее
Postman/Insomnia с сохранением cookies, либо сама SPA — это оговорено
прямо в `info.description` спецификации.

### Авторизация — Sanctum personal access tokens (Bearer-токен)

> Изначально здесь был cookie-session вариант (Sanctum SPA-режим) — в
> процессе разработки я перешёл на Bearer-токены, в первую очередь
> из-за деплоя: cookie-based авторизация требует, чтобы frontend и
> backend жили на одном домене (см. `docs/DEPLOYMENT.md`), а токены
> от этого ограничения свободны. Ниже — как это устроено сейчас.

- `POST /api/auth/login` возвращает `{ data: user, token: "..." }` —
  `AuthController@login` сверяет email/пароль вручную (`Hash::check`,
  без `Auth::attempt()` — он создал бы ещё и сессию, которая тут не
  нужна и раньше приводила к путанице между двумя механизмами auth)
  и выдаёт токен через `$user->createToken()`.
- Токен кладётся в заголовок `Authorization: Bearer <токен>` на каждый
  следующий запрос (`frontend/app/composables/useApi.ts`).
- Хранится токен **в памяти** (`useState`), не в `localStorage` — это
  осознанный компромисс: `localStorage` читается любым JS на странице,
  то есть уязвим для кражи токена через XSS, а `useState` — обычная
  переменная в памяти вкладки, недоступная постороннему скрипту. Цена
  компромисса — обновление страницы (F5) разлогинивает пользователя,
  потому что память вкладки сбрасывается. Для реального продукта это
  стоит закрыть refresh-токеном или httpOnly cookie только для
  refresh-эндпоинта — не реализовано в тестовом, см. «На будущее».
- `POST /api/auth/logout` — `$request->user()->currentAccessToken()->delete()`.
  `auth:sanctum` middleware уже гарантирует, что `$request->user()` не
  `null` к моменту, когда код дойдёт до контроллера — если токен
  отсутствует/невалиден, запрос обрывается ещё на уровне middleware с
  `401`, до `AuthController` дело не доходит.
- CORS больше не требует `supports_credentials: true` (это было нужно
  только для cookie) — `config/cors.php` сейчас проще, чем в
  cookie-варианте.
- `bootstrap/app.php` **не** содержит `statefulApi()` — эта настройка
  специально для cookie-режима; если её оставить рядом с
  Bearer-токенами, Sanctum пытается резолвить пользователя то через
  сессию, то через токен в зависимости от происхождения запроса, и это
  создаёт трудноуловимые баги (именно так родился `500` на logout при
  переходе с cookie на токены — оставшийся `statefulApi()` конфликтовал
  с новым токен-based `AuthController`).

### Права доступа

`GET/POST /api/tasks` доступны любому авторизованному — список
автоматически ограничивается собственными задачами
(`TaskController@index`), для админа — не ограничивается.
`PUT/PATCH/DELETE` защищены `TaskPolicy` (`app/Policies/TaskPolicy.php`)
через `$this->authorize()` — попытка изменить чужую задачу даёт `403`,
а не молчаливый отказ.

### Единый формат ошибок

Вместо `try/catch` в каждом контроллере, весь маппинг исключений в JSON
собран в одном месте — `backend/bootstrap/app.php`
(`withExceptions(...)`): `401` неавторизован, `403` запрещено, `404` не
найдено, `422` ошибки валидации (`{message, errors}`), `500` — всё
остальное, с логированием, но без утечки деталей наружу.

### Frontend

- **`composables/useApi.ts`** — единая точка входа для запросов:
  подставляет `Authorization: Bearer`, редирект на `/login` при `401`.
- **`composables/useAuth.ts` / `useTasks.ts`** — состояние
  пользователя и задач через `useState` (реактивно, без лишней
  библиотеки стейт-менеджмента — для проекта такого размера отдельный
  Pinia избыточен, но добавляется тривиально при росте).
- **`middleware/auth.global.ts`** — route guard: неавторизованного
  редиректит на `/login`, авторизованного не пускает обратно на
  `/login`.
- **`pages/tasks/index.vue`** — список с поиском (debounce 350мс),
  фильтром по статусу, сортировкой (поле + направление), пагинацией,
  состояниями загрузки/ошибки/пустого списка.
- **`components/TaskForm.vue`** — модалка создания/редактирования с
  клиентской валидацией (заголовок 3–255 симв., корректная дата,
  допустимый статус) и показом серверных ошибок валидации построчно.
- Рендеринг — `ssr: false` (чистый SPA). С Bearer-токеном в памяти SSR
  усложняется по другой причине: токен, полученный в браузере,
  недоступен Nuxt-серверу при первом рендере — понадобится либо
  передавать его через отдельный transport (например,
  httpOnly-cookie только для этой синхронизации), либо смириться с
  тем, что первый рендер всегда неавторизован; для тестового задания
  `ssr: false` — осознанное упрощение (см. «На будущее» ниже).

## Проверка ролей/прав руками

Сиды создают трёх пользователей: `demo@example.com` (обычный, есть свои
задачи), `other@example.com` (обычный, тоже есть задачи),
`admin@example.com` (админ). Зайдите как `demo`, попробуйте открыть
задачу `other` по прямой ссылке/API — получите `403`. Зайдите как
`admin` — тот же запрос пройдёт.
